# Audit — 20 ways the verifier doesn't judge the LOOK / visuals correctly

Scope: every channel that's supposed to verify how the build *looks* — `visionChecklistGame` (loop.mjs),
`_gameCanvasDataURL`/`_servedCanvasDataURL` + `screenshotWebGL` (capture), `_gameArtRichness`/`artReview`
(match.mjs), `compare_regions`/`compare_image`, `analyzeStructure` (structure.mjs), and `see_page {visual}`
(Block 69). The theme: it mostly checks **code text** and a **single downscaled still judged by a flaky LLM** —
not what a human actually sees moving on screen.

## A. The vision judge is unreliable
1. **It's an LLM judge — non-deterministic + hallucinates.** `visionChecklistGame` asks gemini to answer
   present:true/false per item. The same render can pass one run and fail the next; it asserts "character
   present" for a blob, or "missing HUD" when the HUD is right there.
2. **The model INVENTS the checklist from the task.** The criteria themselves are generated by the same weak
   judge — a vague/short/wrong checklist verifies the wrong things. There's no fixed, trustworthy rubric.
3. **Skips when the checklist is short.** It only acts when `crit.total >= 4`; a vague task → a 2–3 item list
   → the **entire vision check is skipped → pass**. The fuzzier the request, the less it's verified.
4. **Judges ONE frozen frame — usually the wrong one.** A single screenshot is captured, typically the
   initial/static frame *before* the game starts, before input, before animations. A game that only looks
   right once it's running is judged on its empty start screen.
5. **The capture is downscaled to ≤768px.** `glCaptureInject` shrinks the canvas, so fine detail — textures,
   small sprites, **on-screen text legibility**, thin lines — is gone before the model ever sees it. It can't
   judge detail it wasn't shown.
6. **No motion / animation / "juice" check.** Particles, easing, screen-shake, transitions, frame-rate — none
   are verifiable from a still. "Smooth" vs "janky" is invisible to the verifier.
7. **Degrades to SKIPPED with no key / no browser / no verifyModel.** Offline, no-Chrome, or a non-multimodal
   model → the visual check simply doesn't run, and (pre-Block-78) read as pass. Visuals go unverified silently.
8. **Boolean present/absent, no QUALITY grade.** An item is "present" or not — an *ugly* character, *clashing*
   palette, *misaligned* HUD all count as "present" and pass. Looking-correct ≠ looking-good.

## B. Pixel-match is the wrong tool — or absent
9. **No reference → no pixel check at all.** `compare_regions`/`compare_image` only run when a reference image
   exists. A from-scratch build with no reference gets **zero pixel-level verification** of its look.
10. **≥95% pixel similarity ≠ "looks good".** Region MSE/similarity measures closeness to a target, not
    aesthetic correctness. A blurry, washed-out, or mis-coloured render can still score ≥95%; a crisp,
    better-but-different style fails.
11. **The agent defines the compare REGIONS.** It can draw lenient or empty boxes, skip the hard areas, or
    cover only easy regions — self-grading the very thing being checked.
12. **The reference is AI-slop (design-first).** `generate_image` makes `reference.png`; matching it ≥95% just
    means "looks like our mediocre mockup," not "looks good." A bad generated target enforces a bad result.

## C. "Richness" and "structure" don't measure how it looks
13. **Art-richness is a crude scalar.** `_gameArtRichness < 18` (colour variety / edge density) flags only flat
    boxes. A **noisy random-coloured mess scores HIGH** richness and passes; a clean minimal style scores low
    and is wrongly flagged. Richness ≠ good art.
14. **The richness threshold (18) is arbitrary and low** — it catches only egregious programmer-art; a
    mediocre-but-not-flat render sails through.
15. **`analyzeStructure` reads CODE, not the RENDER.** It greps the source for "enemy/hud/levels/texture". Code
    that *defines* a HUD but renders it **off-screen, invisible, zero-size, or behind other elements** still
    passes the structure gate — it verifies the code mentions it, not that it's visible.
16. **No layout / overlap / z-order / contrast check.** Overlapping sprites, text drawn on a same-colour
    background (invisible), elements off the canvas, a character behind the scenery — none are caught by
    richness or structure, and the flaky vision judge catches them only by luck.

## D. The capture doesn't match what the user sees
17. **Headless render ≠ real browser.** Fonts, WebGL shaders, GPU effects, CSS depending on a real viewport can
    render differently (or fail) headless. The verifier judges a screenshot the user will never actually see.
18. **Screenshot timing is fixed — often too early.** It's captured on a budget, frequently **before images /
    fonts / textures load** or before the first paint settles → a blank/half-loaded look is judged (false
    fail, or the agent "fixes" a non-problem).
19. **One viewport, one state.** It checks a single default size and the default scene. **Responsiveness,
    mobile, dark mode, and the menu / win / lose / pause screens** are never visually verified — only the one
    frame.
20. **Canvas-centric — DOM/SVG/CSS UIs aren't really seen.** Richness + structure + the game vision checklist
    assume a `<canvas>` game. A normal web UI (DOM/CSS), a chart, an SVG, a dashboard gets only a generic
    `see_page` description (if a goal was passed) — no equivalent visual rubric. The look of most non-game web
    work is effectively unverified.

---

## The fixes that would matter most
- **Capture the RUNNING game, multiple frames** (drive a few inputs first, sample 3–5 frames) and judge those
  — not one cold start frame (#4, #6).
- **Don't downscale below legibility** for the vision pass; do a second crop pass on text/detail regions (#5).
- **A fixed, trustworthy rubric** + multiple independent judges with majority vote, instead of one self-derived
  flaky checklist (#1, #2, #3).
- **Deterministic visual lints** the LLM can't fudge: detect invisible/same-colour-on-background elements,
  off-canvas draws, zero-size HUD, text-contrast — render-level, not code-level (#15, #16).
- **Fail CLOSED**: a skipped visual check (no key/browser/short checklist) is "unverified," never "looks good"
  (#3, #7).
- **A non-game visual rubric** for DOM/CSS/SVG UIs (layout sanity, contrast, overflow) (#20).
